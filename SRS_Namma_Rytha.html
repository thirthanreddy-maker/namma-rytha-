1Introduction
1.1 Purpose

This Software Requirements Specification (SRS) document describes the complete functional and non-functional requirements for Namma Rytha (ನಮ್ಮ ರೈತ), an AI-powered smart farming platform designed for Indian farmers. The document is intended for use by developers, testers, academic evaluators, and stakeholders to understand the system's objectives, features, constraints, and design.

This SRS conforms to the IEEE 830-1998 standard for software requirements specifications and serves as the authoritative reference throughout the software development lifecycle (SDLC).

1.2 Scope

Namma Rytha is a full-stack web and mobile application that empowers Indian farmers with AI-driven agricultural intelligence. The platform integrates real-time weather data, precision irrigation planning, fertilizer prescription engines, crop advisory systems, plant disease detection, live market prices, a sustainability tracker, and an online marketplace for agricultural tools.

The system supports two primary user roles: Farmers (end users) and Administrators (platform managers). It is accessible via any modern web browser and as a Progressive Web App (PWA) / native Android application through Capacitor.

✅ What is in scope: Web application, REST API backend, SQLite database, Android APK build, AI-powered recommendations via Google Gemini, and real-time weather integration.
⚠️ What is out of scope: IoT hardware integration, real-time soil sensor connectivity, iOS native build, payment gateway integration, SMS/email OTP delivery (currently demo mode).

2Overall Description
2.1 Product Perspective

Namma Rytha is a standalone full-stack application consisting of three tiers:

  • Presentation Layer: HTML5/CSS3/JavaScript SPA served as static files, also packaged as a PWA and Android APK via Capacitor.
  • Business Logic Layer: Node.js + Express.js RESTful backend deployed on Render.com (cloud) or run locally on port 3000.
  • Data Layer: SQLite database with tables for users, admins, products, orders, farm data, feedback, and activity logs.

External integrations include:

  • Google Gemini 2.0 Flash — generative AI for farming advice and chat.
  • Open-Meteo API — free, real-time weather and 7-day forecast.
  • Nominatim (OpenStreetMap) — geocoding for location-based weather.
  • Google OAuth 2.0 — secure Google Sign-In for farmers and admins.
  • Leaflet.js + Globe.gl — interactive farm maps.

3.3 Data Flow
  1. User opens the app → index.html checks localStorage for session token.
  2. If unauthenticated → redirected to login.html.
  3. On login → credentials sent via POST to /api/login or /api/google-auth.
  4. Successful response stored in localStorage as nr_user.
  5. Dashboard loads → real-time weather fetched from Open-Meteo, farm data from backend.
  6. AI modules process local logic offline; complex queries are sent to Google Gemini API.
  7. Activities are logged asynchronously to /api/activity for admin monitoring.
4Functional Requirements
4.1 Authentication Module
  • FR-AUTH-01The system shall allow farmers to register with first name, last name, email, phone, location, land area (acres), primary crop, and password.
  • FR-AUTH-02The system shall validate that the email address is unique before creating an account.
  • FR-AUTH-03The system shall allow farmers to log in with email and password.
  • FR-AUTH-04The system shall support Google Sign-In via OAuth 2.0; if the Google email already exists, the account shall be linked automatically.
  • FR-AUTH-05The system shall provide a password-reset flow: generate a 4-digit OTP (valid for 10 minutes), verify OTP, and update password.
  • FR-AUTH-06The system shall maintain session state via localStorage and automatically redirect unauthenticated users to the login page.
  • FR-AUTH-07Admins shall register using a secure invite code; Google Sign-In for admins shall only succeed if the email is pre-registered in the admins table.
  • FR-AUTH-08Mobile users (screen width ≤ 768px or Capacitor native) shall be automatically redirected to the mobile-optimised view.
4.2 Dashboard Module
  • FR-DASH-01The dashboard shall display real-time KPI cards: Soil Moisture (%), Temperature (°C), Rain Probability (%), and Soil pH.
  • FR-DASH-02The dashboard shall display a 5-day weather forecast panel fetched from Open-Meteo.
  • FR-DASH-03The dashboard shall show AI-generated farming recommendations refreshed on demand.
  • FR-DASH-04The dashboard shall display an active alerts panel with critical, warning, and informational alerts. Alerts shall be individually dismissible.
  • FR-DASH-05The dashboard shall provide Quick Action buttons for Irrigate Now, Fertilizer Plan, Find Best Crop, Check Disease Risk, Market Prices, and Ask AI.
  • FR-DASH-06The dashboard shall display a mini sustainability score (Water Saved, Eco Score, Yield Index) with links to the full tracker.
  • FR-DASH-07The dashboard shall show an Agri-News & Market Trends horizontal slider with market watch, weather insights, and government policy cards.
  • FR-DASH-08The user shall be able to export a PDF farm report and share it via WhatsApp from the top bar.
4.3 Smart Irrigation Advisor
  • FR-IRR-01The system shall accept inputs: soil moisture (%), crop type, rain expected (yes/no), temperature (°C), last irrigation (days ago), and field area (acres).
  • FR-IRR-02The system shall compute irrigation decisions: SKIP (rain/moisture sufficient), IRRIGATE NOW (critical moisture), IRRIGATE SOON (sub-optimal), NO IRRIGATION NEEDED.
  • FR-IRR-03The system shall compute water required (litres) based on crop type, moisture deficit, and field area.
  • FR-IRR-04The result shall include: action recommendation, detail explanation, water saved percentage, next irrigation date, and actionable tips.
  • FR-IRR-05The irrigation form shall auto-populate soil moisture and temperature from the current dashboard live values when the tab is opened.
  • FR-IRR-06The system shall save the farm data (moisture, rain probability, timestamp) to the backend after each analysis.
  • FR-IRR-07The system shall update the sustainability telemetry (water saved metric) after each irrigation analysis.
4.4 Fertilizer Recommendation Engine
  • FR-FERT-01The system shall accept: soil pH (4–9), nitrogen (N) kg/acre, phosphorus (P) kg/acre, potassium (K) kg/acre, crop type, and soil type.
  • FR-FERT-02The system shall compute the NPK deficit against crop-specific ideal requirements for Wheat, Rice, Tomato, Cotton, Sugarcane, and Maize.
  • FR-FERT-03The system shall prescribe specific fertilisers: Urea (46% N), Super Phosphate (16% P₂O₅), Muriate of Potash (60% K₂O), and Zinc Sulphate as a micronutrient.
  • FR-FERT-04The system shall generate a soil correction recommendation based on pH: lime for acidic (<5.5), gypsum for alkaline (>7.8).
  • FR-FERT-05The system shall generate a crop-specific application schedule (basal, split-dose timings).
4.5 Crop Advisor
  • FR-CROP-01The system shall accept: soil type, season (Kharif/Rabi/Zaid), average temperature, annual rainfall (mm), soil pH, and water availability level.
  • FR-CROP-02The system shall match inputs against a database of 10 crops (Wheat, Rice, Tomato, Cotton, Soybean, Sugarcane, Maize, Onion, Groundnut, Turmeric) using a weighted scoring algorithm.
  • FR-CROP-03The system shall display the top 5 matching crops ranked by match percentage (max 98%) with medals (🥇🥈🥉) and reason tags.
  • FR-CROP-04The season selector shall auto-detect the current month and pre-select the appropriate season (Kharif: Jun–Oct, Rabi: Nov–Feb, Zaid: Mar–May).
  • FR-CROP-05The system shall include a Yield Predictor that estimates total yield (quintals) and estimated revenue (₹) based on crop, area, and season quality.
4.6 Disease Detector
  • FR-DIS-01The system shall allow the user to select a crop and check one or more visible symptoms from a symptom grid.
  • FR-DIS-02The system shall diagnose diseases including: Powdery Mildew, Late Blight, Fusarium Wilt, Bacterial Leaf Blight, Brown Rust, Root Rot, and Yellow Mosaic Virus.
  • FR-DIS-03Each diagnosis shall include: disease name, description, risk level (Low/Medium/High), treatment plan (chemical + organic), and prevention strategies.
  • FR-DIS-04The system shall display a Disease Risk Calendar showing historical seasonal risk patterns for the selected crop.
  • FR-DIS-05Humidity from the live weather data shall auto-populate the disease input form when the tab is opened.
4.7 Weather Intelligence
  • FR-WX-01The system shall fetch real-time weather using the Open-Meteo API for the user's current GPS location (via browser Geolocation API) or a manually entered city.
  • FR-WX-02Current conditions displayed shall include: temperature, apparent temperature (feels like), humidity, wind speed, UV index, weather icon, and rain probability.
  • FR-WX-03The system shall display a 7-day forecast with daily high/low temperatures, weather icon, and max precipitation probability.
  • FR-WX-04The system shall derive rain probability for the next 12 hours from hourly data and update the dashboard rain card.
  • FR-WX-05A topbar weather pill shall always display the current temperature and icon for quick reference.
4.8 Market Price Module
  • FR-MKT-01The system shall display current mandi prices for major crops (Wheat, Rice, Tomato, Cotton, Onion, Soybean, Maize, Sugarcane, Potato) in ₹/quintal.
  • FR-MKT-02Prices shall be filterable by state (Karnataka, Maharashtra, Punjab, Uttar Pradesh, Andhra Pradesh, Tamil Nadu).
  • FR-MKT-03Each price entry shall show the trend (up/down), MSP (Minimum Support Price), and a best-time-to-sell recommendation.
  • FR-MKT-04The system shall display an AI-powered "Price Forecast" section indicating expected price movement for the next 30 days.
4.9 Sustainability Tracker
  • FR-SUS-01The system shall maintain a sustainability score (0–100) per user, persisted to the backend database.
  • FR-SUS-02The score shall be computed from: water saved (litres), irrigation checks, fertilizer checks, disease checks, and AI usage events.
  • FR-SUS-03The system shall display animated circular progress indicators for Water Saved, Eco Score, and Yield Index.
  • FR-SUS-04The system shall maintain an achievements system with 6 badges (Water Warrior, Green Thumb, Data Farmer, Gold Farmer, Earth Guardian, Super Farmer).
  • FR-SUS-05The system shall display a telemetry breakdown of total water saved (litres), chemicals reduced (kg), CO₂ equivalent avoided, and AI-based decisions made.
4.10 Marketplace
  • FR-MKP-01The system shall display products fetched from the backend API, categorised into: Tractors & Equipment, Irrigation & Water, Hand Tools, Sprayers, Harvesting, Smart Accessories, Safety & PPE.
  • FR-MKP-02Products shall display: name, category, price (₹), description, brand, rating (stars), and a suitable crop/soil filter tag.
  • FR-MKP-03The system shall support category filtering, price sorting (low-high, high-low), and a product search box.
  • FR-MKP-04Users shall be able to add products to a shopping cart and a wishlist. Cart and wishlist counts shall be displayed in the topbar.
  • FR-MKP-05Users shall be able to place an order; the order shall be saved to the backend orders table with status "pending".
4.11 Admin Panel
  • FR-ADMIN-01The admin panel shall display summary KPI cards: Total Farmers, Active Users (last 24h), Total Products, and Pending Orders.
  • FR-ADMIN-02The admin shall be able to view, search, and delete farmer accounts from the Farmers Management table.
  • FR-ADMIN-03The admin shall be able to add, edit (name, category, price, description), and delete products from the Products Management table.
  • FR-ADMIN-04The admin shall be able to view all recent orders with user name, items, total, and order status.
  • FR-ADMIN-05The admin shall be able to view an activity log showing all farmer actions (page views, analyses, orders) with timestamps.
  • FR-ADMIN-06The admin panel shall support Google Sign-In; only pre-registered admin emails shall be allowed to log in via Google.
4.12 AI Chat Advisor
  • FR-AI-01The system shall provide a floating chat interface accessible from any page via a chat button.
  • FR-AI-02Chat shall be powered by Google Gemini 2.0 Flash API, with a system prompt that positions the AI as an expert Indian agricultural advisor.
  • FR-AI-03The system shall maintain a multi-turn conversation history (chat history) within the current session.
  • FR-AI-04Context-sensitive "Ask AI" buttons throughout the app shall pre-fill the chat with relevant field data (moisture, crop, symptoms, etc.).
  • FR-AI-05The system shall fall back to curated offline responses if the Gemini API key is not configured or the request fails.
  • FR-AI-06Voice input shall be supported via the Web Speech API for hands-free query entry.
4.13 Multilingual Support
  • FR-I18N-01The system shall support 6 languages: English (default), Kannada, Hindi, Telugu, Tamil, and Marathi.
  • FR-I18N-02Language preference shall be persisted in localStorage and applied on every page load.
  • FR-I18N-03UI elements marked with data-i18n attributes shall be dynamically translated using the TRANSLATIONS dictionary.
  • FR-I18N-04A language switcher dropdown in the topbar shall allow users to switch language at any time without reloading the page.
5Non-Functional Requirements
⚡ Performance

Initial page load shall complete in <3 seconds on a 4G connection (≥10 Mbps).

AI analysis results (irrigation, fertilizer, crop) shall be rendered within 1 second (local computation).

Gemini API responses shall appear within 5 seconds under normal network conditions.

Backend REST endpoints shall respond in <200ms for database reads under normal load.

🔒 Security

Passwords shall be stored as plaintext only in the current academic prototype; production deployment MUST use bcrypt hashing.

Admin access is gated by invite code; Google OAuth admin login is restricted to pre-registered emails.

CORS is enabled; API endpoints shall validate required fields and return 400/401/500 HTTP codes.

OTPs expire after 10 minutes and are invalidated after single use.

📱 Usability

The UI shall be fully responsive — optimised for desktop (≥1024px), tablet (768–1023px), and mobile (<768px).

Mobile users shall be automatically redirected to the mobile-optimised view.

The interface shall provide visual feedback (toast notifications, loading spinners) for all asynchronous operations.

WCAG 2.1 AA contrast ratios shall be maintained throughout the UI.

🌐 Availability

Core AI analysis features (irrigation, fertilizer, crop, disease) operate entirely offline after initial load.

Cloud deployment on Render.com free tier may experience cold-start latency of up to 30 seconds after inactivity.

Target uptime: ≥99% for locally hosted deployment.

Service Worker (sw.js) caches static assets for offline PWA functionality.

🔧 Maintainability

All API endpoints are self-contained in server.js with consistent response formats.

Frontend logic is modular — each feature (irrigation, fertilizer, etc.) is a self-contained function set in app.js.

Translation strings are centralised in translations.js for easy language additions.

📈 Scalability

SQLite is suitable for up to ~1,000 concurrent users; for production scale, migration to PostgreSQL is recommended.

The stateless REST API design supports horizontal scaling behind a load balancer.

Product catalog and user data are unbounded by schema and support any number of entries.

9Risk Analysis
#RiskCategoryLikelihoodImpactMitigation Strategy
R-01 Gemini API key quota exhausted or revoked External Medium High Implement offline fallback responses. Allow user to configure their own API key in Settings.
R-02 Open-Meteo or Nominatim API unavailable External Low Medium Fall back to user-entered location. Cache last-known weather in localStorage.
R-03 Render.com free-tier cold start delay (up to 60s) Infrastructure High Medium Show loading animation. Recommend paid Render tier for production. Local deployment avoids this.
R-04 SQLite database file corruption or data loss Data Low High Regular database backups. For production, migrate to PostgreSQL with replication.
R-05 Password stored as plaintext (security vulnerability) Security High High Implement bcrypt hashing before production deployment. Add JWT-based session management.
R-06 Admin invite code exposed in client-side code Security Medium Medium Validate invite code server-side only (currently done). Move to environment variable.
R-07 Inaccurate AI recommendations mislead farmers Functional Medium High Add disclaimer text. Show recommendations as advisory only. Encourage consulting local KVK.
R-08 Browser geolocation denied by user UX Medium Low Provide city-name search as fallback. Persist last location in localStorage.
